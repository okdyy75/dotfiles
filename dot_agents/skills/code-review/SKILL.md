# コードレビュー

修正内容に対して、コードレビューを実施してください。

## レビュー観点

- **一貫性** — 既存の実装と一貫性があるか確認する。同じディレクトリ・同じレイヤーの既存コードと命名規則・構造・パターンが揃っているか確認する。不一致がある場合は既存側のコードを具体的に示すこと
- **新規概念** — 新しいパターンや抽象化（ConcernsやServiceクラス）など既存の仕組みに無い概念やルールが追加されている場合はレビュー指摘に上げること
- **過度な共通化** — 過度な共通化やDRYを意識した実装になっていないこと。シンプルさを優先する
- **チェックリスト**: チェックリスト項目の違反


## レビューチェックリスト

### 共通
- [ ] マジックナンバーやハードコードされた値を使用していないか
- [ ] 不要なデバッグログやコメントが残っていないか

### バックエンド（Ruby / GraphQL）
- [ ] Mutationの引数はInputObjectに切り出しているか
- [ ] Mutation / Rakeタスク / job 内のビジネスロジックをModelに適切に切り出しているか
- [ ] DBに保存されているxxx_at（日時）はUTCで、xxx_date（日付）はJSTという前提で実装されているか

### フロントエンド（TypeScript / React）
- [ ] 無駄なデフォルト引数やオプショナル引数を許可していないか
- [ ] ロジックはcomponentsファイルに直接書かず、hooksファイルに切り出しているか
- [ ] React Hook Form + zodでバリデーションしているか（フォームがある場合）
- [ ] `useCallback` や `useMemo` を使用していないか（原則使用禁止）
- [ ] margin や padding に頼らず `spacing` や `gap` を使用しているか
- [ ] `reset()` の値セットで不要な `?? undefined` を指定していないか（APIレスポンスの型をそのまま使用する）


## レビュー手順

1. 変更ファイルごとに、同ディレクトリ・同レイヤーの既存コードを読み取り、既存の命名規則・構造・パターンを把握する
2. レビューチェックリストに基づいてレビューを実施する
3. レビュー観点に基づいてレビューを実施する
4. 結果を出力


## 出力フォーマット

下記二つの項目について出力してください

1. レビューチェックリストの結果をそのまま出力

2. 指摘がある場合は以下の形式で出力する

```
### レビュー結果

| # | 優先度 | ファイル | 行 | 種別 | 指摘内容 |
|---|---|---------|-----|------|---------|
| 1 | P1 | path/to/file.rb | 42 | 一貫性 | 既存のXXXと命名規則が異なる |
| 2 | P2 | path/to/file.tsx | 15 | チェックリスト | useMemoが使用されている |

#### 詳細

**1. path/to/file.rb:42 — 一貫性**
（具体的な説明と改善案）

**2. path/to/file.tsx:15 — チェックリスト**
（具体的な説明と改善案）
```
